# Miami-Dade County Open Data Encyclopedia

## What This Is

A CLI tool and static encyclopedia that catalogs, AI-enriches, and serves every dataset from South Florida's ArcGIS Open Data Hubs as a searchable website. Residents, journalists, and civic hackers can browse, search, and understand what data Miami-Dade County, Broward County, and City of Miami publish — with plain-English descriptions, quality scores, AI-generated column descriptions, and practical use cases generated by Claude. Includes subscribable change feeds and downloadable catalog exports. Deployed to GitHub Pages with weekly auto-refresh.

## Core Value

Every South Florida open dataset is discoverable and understandable by a non-technical resident.

### Out of Scope

- Data preview (showing actual rows) — extra API cost, source portal already provides this
- Real-time data — this is a catalog, not a data warehouse
- OAuth or user accounts — pure static site, no auth needed
- Mobile app — responsive static site works well on mobile
- Email/Slack notifications — requires notification infrastructure and user accounts
- Hosted API for programmatic access — conflicts with static site architecture; catalog export addresses developer needs
- Comments/discussion on datasets — requires moderation and backend
- Embeddable dataset card widgets — v2 consideration
- Real-time feed updates (WebSub/PubSubHubbub) — static site, weekly cadence makes this irrelevant
- DCAT-US v3.0 compliance — draft spec, use v1.1 field names and migrate when finalized

## Context

**Shipped v1.1** with 7,298 LOC across Python (4,861), HTML/CSS/JS (2,260), and YAML (177).

**Tech stack:** Python 3.12+, SQLite (Schema V4), Click/rich-click CLI, Jinja2 templates, Anthropic Python SDK (structured output for field enrichment), Pydantic, feedgen, Lunr.js client-side search, vanilla HTML/CSS/JS, GitHub Actions, GitHub Pages.

**Data sources:** Three ArcGIS Open Data Hubs:
- Miami-Dade County: `gis-mdc.opendata.arcgis.com` (570+ datasets)
- Broward County: `gis-broward.opendata.arcgis.com`
- City of Miami: `gis-miami.opendata.arcgis.com`

**Outputs:** Static HTML site, Atom 1.0 feed, DCAT-US JSON catalog, Excel CSV catalog.

**Known tech debt:**
- browse.html grade CSS class mismatch (cosmetic, one-line fix, carried from v1.0)
- Push-trigger deploys empty site before first cron run (self-resolving)
- SUMMARY.md frontmatter missing `requirements-completed` field (non-blocking)

## Constraints

- **Rate limiting**: ArcGIS Hub rate-limited to 1 request/second
- **Static site**: Must work without a server — pure static files, no backend at runtime
- **Cost transparency**: AI enrichment must show cost estimate and support dry-run before spending API credits
- **Packaging**: Installable via `pip install .` with `pyproject.toml` entry point for `mdc-encyclopedia` CLI
- **Tech stack**: Python 3.12+, SQLite, Click CLI, Jinja2 templates, Anthropic Python SDK, Rich terminal output, vanilla HTML/CSS/JS
- **Multi-jurisdiction**: All features must work across all registered jurisdictions without hardcoded assumptions

## Key Decisions

| Decision | Rationale | Outcome |
|----------|-----------|---------|
| SQLite over Postgres | Single-file DB, no server needed, works in CI and locally | ✓ Good — DB committed to repo for CI persistence |
| Lunr.js for search | Client-side search avoids backend, works with static hosting | ✓ Good — field boosts (title 10x, keywords 8x) give relevant results |
| GitHub Pages deployment | Free hosting, integrates with Actions for weekly refresh | ✓ Good — artifact-based deploy, no gh-pages branch needed |
| No data preview | Avoids extra API calls per dataset, keeps scope focused on discovery | ✓ Good — source portal links serve this need |
| ArcGIS Hub only (no Socrata) | Research confirmed Miami-Dade has no active Socrata portal | ✓ Good — simplified client code, one API to maintain |
| PRAGMA user_version for schema | Lightweight versioning without extra tables | ✓ Good — V1→V4 migrations all worked cleanly |
| Synchronous httpx (not async) | 1 req/s rate limit makes async pointless | ✓ Good — simpler code, same performance |
| DB committed to repo | actions/cache evicts after 7 days, too risky for data persistence | ✓ Good — [skip ci] prevents infinite loops |
| YAML for jurisdiction config | Supports comments, more human-editable than JSON | ✓ Good — clean registry file |
| Synthetic composite key (jurisdiction_arcgis_id) | Preserves single-column FK pattern, avoids multi-column joins | ✓ Good — zero data loss in V3 migration |
| Denormalized ai_description on datasets | Faster queries, avoids JOIN to enrichments for common field | ✓ Good — write-through in insert_enrichment |
| Template-driven enrichment prompts | Single template with `{jurisdiction}` variable, no per-jurisdiction prompt files | ✓ Good — scales to new jurisdictions via registry |
| feedgen for Atom feeds | Mature library, handles XML escaping and spec compliance | ✓ Good — valid Atom 1.0 output |
| ext_ prefix for DCAT-US extensions | Avoids namespace collisions with standard DCAT fields | ✓ Good — clear convention |
| Enrichment gated on workflow_dispatch | Prevents accidental API spend on scheduled CI runs | ✓ Good — manual opt-in via enrich_ai boolean |
| INSERT OR REPLACE with subquery for upsert | Mirrors upsert_dataset pattern, preserves ai_description during re-pull | ✓ Good — closes data loss tech debt |
| Orphan column deletion | Matches API truth rather than keeping stale column records | ✓ Good — clean column data |
