## Cross-Phase Integration

### Connected Exports (12/14)

All Phase 8 exports properly wired within the pull pipeline:
- `load_registry()` → `cli.py` pull command
- `create_client(base_url)` → per-jurisdiction in CLI loop
- `normalize_hub_dataset(jurisdiction, hub_url)` → synthetic ID generation
- `_migrate_v2_to_v3()` → `init_db()` auto-migration
- `_backup_database()` → called before migration SQL
- `upsert_dataset()` → V3-compatible with jurisdiction/arcgis_id columns

### Missing Connections (2)

**1. ai_description write-through gap**
- `insert_enrichment()` writes to `enrichments.description` but never updates `datasets.ai_description`
- Column populated only during V3 migration from pre-existing enrichment data
- Site works today via JOIN alias (`e.description AS ai_description` in `context.py`)
- Phase 10 exports may read `datasets.ai_description` directly and find NULL
- Affected: MULTI-03, EXPORT-01, EXPORT-02

**2. Enrichment jurisdiction context not wired**
- `build_system_prompt()` built in `prompts.py` but not called by `enrich_dataset()` in `client.py`
- All enrichments use hardcoded Miami-Dade `SYSTEM_PROMPT` regardless of dataset jurisdiction
- Documented as deferred to Phase 11 in 08-02-SUMMARY.md
- Affected: MULTI-02 (enrichment path only), FIELD-01

### Broken E2E Flows (1)

**Enrich non-Miami-Dade datasets** — Broward/Miami datasets get Miami-Dade department taxonomy in AI enrichments. Breaks at `client.py` line 71 (`system=SYSTEM_PROMPT`). Fix planned for Phase 11.
